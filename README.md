## 一、这是什么

这不是一个普通的 Obsidian 主题或插件,这是一套**已经在 5000+ 笔记的真实知识库里跑了一年**的方法论 + 配套 Skill。

### 解决什么问题

当 Obsidian 知识库膨胀到 1000+ 文件后,会出现典型的"知识沼泽"症状:

- 每天自动抓取的推特/公众号原文淹没真正有用的精炼笔记
- 主题分散在十几个目录里,找一个东西要打开 N 个文件夹
- 概念笔记重复,比如"知识库管理"建了三个节点
- 双向链接(Wiki Link)失修,无法把相关知识串起来
- 检索时大量噪声进入上下文,让 AI 助手回答质量下降
- **(v3 新发现)个人原话/亲历/感受被 SOP 萃取流程"磨平",失去 AI 时代最稀缺的"你"**

### 七大解决机制

| 机制 | 解决什么 |
|------|---------|
| **1. 4+1 层信息分层(v3)** | 个人经验 / 在场素材 / 入口 / 萃取成品 / 冷藏 物理隔离 |
| **2. PARA 目录骨架** | Projects / Areas / Resources / Archive 四象限 |
| **3. P.A.I.R 收录顺畅度** | 先收再理,Inbox 缓冲层 |
| **4. 冷藏区隔离** | 自动抓取的低密度原料独立目录,主区检索默认豁免 |
| **5. 命名前缀规范** | `[MOC]` / `[理论]` / `[工程]` / `[方法论]` 让概念节点一眼分类 |
| **6. 高频场景路由表(v3)** | 12 个高频场景 → 对应目录,AI 助手按表自动路由 |
| **7. Wiki Link 自动识别(v3.1)** | 新建/更新文件后 8 维实体扫描 + 7 步建立链接,告别死链失修 |

## 二、核心思想:为什么这么做

完整思想文档见 [PHILOSOPHY.md](./PHILOSOPHY.md)。这里只列骨架。

### 一句话总结

> **AI 上下文越长,越要从源头分流。**
>
> 真正稀缺的不是上下文长度,是上下文质量。
> 把不同加工程度的信息物理隔离,不让低密度原料淹没你的一手感受。

### 四个思想来源

| 来源 | 提出者 | 解决什么 | 我们怎么用 |
|------|-------|---------|-----------|
| **PARA** | Tiago Forte《Building a Second Brain》 | 目录结构 | `01-Projects` ~ `07-Archive` 骨架 |
| **P.A.I.R** | flomo / 少楠 | 收录顺畅度 | `00-Inbox/` 缓冲机制 |
| **LLM Wiki** | Andrej Karpathy(2026 公开分享) | AI 时代复利 | `04-Concepts/` 概念枢纽 + AI 维护 |
| **4 层信息分层** | 余一(2026-05 公开分享) | 信息源头污染 | 引出 4+1 层模型 |

### 4+1 层信息分层模型(v3 核心)

| 层 | 名称 | 定义 | 物理目录 | frontmatter |
|---|------|-----|---------|-------------|
| **L1** | 个人经验层 | 用户原话/亲历/一手感受 | `02-Areas/个人经验/` | `layer: experience` |
| **L2** | 在场素材层 | 用户在场录音/会议原文 | `07-Archive/播客转录/` | `layer: scene` |
| **L3** | 外部入口层 | 待分类原料 | `00-Inbox/` | `layer: inbox` |
| **L4** | 萃取成品层 | 加工后的 SOP / 概念 / 方法论 | `02-Areas/04/05/06` | `layer: extract` |
| **L5** | 冷藏原料层 | 自动抓取的低频原料 | `99-冷藏/` | `layer: cold` |

**关键洞察**:L1 个人经验是 AI 永远学不会的"你",必须用目录物理隔离,绝不让"萃取冲动"污染原汁原味的一手感受。L1 跟创业日记/Playbook 的区别就在于——日记是为了萃取 SOP(L4),个人经验是保留原汁原味(L1)。

## 三、知识库框架:目录/分层/命名/链接

### 3.1 主区目录结构(默认检索)

| 目录 | 定位 | 收录什么 |
|------|------|---------|
| `!_MOC/` | 导航中心 | 总入口、画像、知识库总览(`!` 前缀置顶) |
| `00-Inbox/` | 收件箱(L3) | 不确定分类的新内容 |
| `01-Projects/` | 当前项目(L4) | 有明确交付物的进行中项目 |
| `02-Areas/<领域>/` | 长期领域(L4) | 按领域分子目录(用户自定义) |
| `02-Areas/个人经验/` | **L1 个人经验层(v3)** | 原话/感受/亲历;含 3 个子目录 |
| `02-Areas/<knowledge-meta>/` | 元方法论 | 关于知识库本身的方法论 |
| `03-People/<同行/客户/导师/合作/团队>/` | 人物库 | 各类人物档案 |
| `04-Concepts/` | 概念枢纽(L4) | 跨域核心概念(命名带前缀) |
| `05-Playbooks/` | SOP 与方法论(L4) | 各类可复用方法论 |
| `06-Library/<工具/失败/书籍/论文/网络/课程>/` | 资源库(L4) | 工具清单、失败案例、读书笔记等 |
| `07-Archive/已结案项目/` | 归档 | 完成的项目记录 |
| `07-Archive/播客转录/` | 归档(高价值 · L2) | 用户在场播客原文 |
| `07-Archive/论文原文/` | 归档(高价值) | 论文原文存档 |

### 3.2 L1 个人经验层细分(v3 新增)

| 子目录 | 准入规则 | event_type |
|--------|---------|------------|
| `原话与决策/` | 用户亲口说出的话、重大决策原话、对外正式表态 | `原话` |
| `感受与判断/` | 旁观者观察到的用户情绪、直觉判断、价值观显形 | `感受` |
| `亲历事件/` | 现场记录的事件,不带 SOP 萃取意图 | `亲历` |

### 3.3 冷藏区(默认不检索)

| 目录 | 定位 |
|------|------|
| `99-冷藏/推特/` | 推特抓取目标 |
| `99-冷藏/公众号博主库/` | 公众号自动抓取 |
| `99-冷藏/朋友圈素材/` | 社交媒体素材 |
| `99-冷藏/活动文案/` | 活动文案、广告文案 |

> 高价值原料(精读公众号文章、精选论文)保留在主区 `07-Archive/`,不并入冷藏。

### 3.4 命名前缀规范(04-Concepts)

| 前缀 | 含义 | 示例 |
|------|------|------|
| `[MOC]` | Map of Content,多文件汇总索引 | `[MOC] 用户分层与定价体系.md` |
| `[理论]` | 概念定义、原理、本质 | `[理论] 知识库与RAG完全指南.md` |
| `[工程]` | 具体实现、技巧、最佳实践 | `[工程] 稀缺性推理在上下文工程中的应用.md` |
| `[方法论]` | SOP 之上的元方法论 | `[方法论] 知识库管理.md` |

**冗余词黑名单**(节点名禁用):调研、报告、深度、完全、详解、全景。

**英文节点**:除专有名词(NVIDIA / API / Harness / Karpathy)外,全部翻译成中文,专有名词附中文说明。

### 3.5 frontmatter 标准(v3 强制 layer 字段)

```yaml
---
tags: [主标签, 辅助标签]
created: YYYY-MM-DD
layer: extract  # 必填:experience / scene / inbox / extract / cold
source: <URL 或来源说明>
author: <作者>
cherry_picked: false  # 仅冷藏区淘金后才设 true
---
```

L1 个人经验层额外字段:

```yaml
event_date: YYYY-MM-DD
event_type: 原话 / 感受 / 亲历
```

### 3.6 高频场景路由表(v3 新增,共 12 场景)

| 场景 | 候选目录(优先级) |
|------|-----------------|
| 客户/商务/定价 | `03-People/客户档案/` → `05-Playbooks/<sales>/` → `01-Projects/` |
| 内容创作/选题 | `02-Areas/内容创作/` → `05-Playbooks/<content>/` → `04-Concepts/` |
| 产品/程序开发 | `01-Projects/` → `02-Areas/<engineering>/` → `06-Library/工具清单/` |
| AI/Agent 理论 | `04-Concepts/[理论] <主题>` → `02-Areas/<ai-domain>/` |
| AI 行业动态 | `02-Areas/<ai-industry>/` → `99-冷藏/公众号博主库/` |
| 创业/团队/战略 | `05-Playbooks/<startup>/` → `02-Areas/<strategy>/` |
| **个人原话/复盘/感受** | `02-Areas/个人经验/` → `05-Playbooks/<diary>/`(萃取后) |

### 3.7 Wiki Link 三层结构 + 自动识别(v3.1 升级)

每篇精炼笔记必须建立三类链接:

| 层级 | 类型 | 形式 |
|------|------|------|
| **向上链接** | 链接到所属领域的 MOC | `[[_MOC-<领域>]]` |
| **横向链接** | 链接到相关概念节点 | `[[[前缀] 概念名]]` |
| **向下链接** | 链接到引用的具体文档 | `[[YYYY-MM-DD-具体记录]]` |

**v3.1 关键升级**:新建/更新文件后,助手必须按 8 维实体扫描自动识别候选 wikilink。

8 维扫描:跨域概念 / 人物档案 / 项目交付物 / 长期领域 / 方法论 SOP / 工具 / 课程书籍论文 / L1 个人经验。

**链接质量门槛**:

| 文件类型 | wikilink 数量 |
|---------|--------------|
| `04-Concepts/` 节点 | ≥ 5,至少 3 跨域 |
| `02-Areas/` 文件(L4) | ≥ 2 |
| `02-Areas/个人经验/` L1 文件 | ≥ 1 |
| `06-Library/` 工具清单 | ≥ 1 |

完整 wikilink 7 步 SOP 见 [SKILL.md § Wiki Link 自动识别与维护](./SKILL.md#wiki-link-自动识别与维护v31-升级--强制)。

## 四、Skill 逻辑框架:触发/流程/原则/禁止

### 4.1 触发时机

Skill 在以下任一情况自动加载:

- 用户说"收录到知识库"、"存到 Obsidian"、"入库"
- 用户提问需要检索知识库
- 整理/归类/移动知识库文件
- 补充或维护 Wiki Link、双向链接、加链接、补链接(v3.1)
- 创建任何新笔记(v3.1 强制扫描 wikilink)
- 创建或更新 MOC 索引
- 从冷藏区淘金提炼
- 记录"个人经验"(原话/亲历/感受/判断)→ 走 L1 流程

### 4.2 6 大工作流

```
1. 收录流程   → 类型判断 → 选目录 → frontmatter → 创建笔记 → 8 维 wikilink 扫描 → 更新 MOC
2. 检索流程   → 目录路由(高频场景表)→ 关键词 Grep → Wiki Link 跳转
3. 淘金流程   → 99-冷藏 grep → 评估 → 复制(不剪切)→ 主区版本加 cherry_picked
4. 回写流程   → 综合性分析 → 判断归属 → 创建/追加 → 标注来源 → 补 Wiki Link
5. 整理流程   → Inbox 周清理 + 冷藏季度淘金 + Archive 持续提炼
6. Wikilink 自动识别(v3.1)→ NER 提取 → vault 内 Glob/Grep → 命中分级 → 追加 [[]] → 反向链接 → MOC 同步 → 质量验证
```

### 4.3 6 条核心原则

1. **先收再理(P.A.I.R)**:分类不确定 → 先扔 `00-Inbox/` → 定期归位
2. **原料与成品分离(PARA + 4+1 层)**:L1/L2/L5 是原料,L4 是成品,L3 是缓冲
3. **项目有终点,领域没终点**:Projects 完成后 → SOP 提到 Playbooks,记录归 Archive
4. **原文神圣不可侵犯**:原文一字不改,助手分析追加在独立章节
5. **复利优先**:每次入库都问"能不能复用?"能 → 同步写入 Playbooks/Concepts
6. **冷藏不删除**:冷藏区是隔离不是丢弃,高价值发现复制到主区,源文件加 `cherry_picked: true`

### 4.4 10 条禁止事项

1. 不删除任何文件(除非用户明确要求)
2. 不修改原文内容(只追加,原文一字不改)
3. 不用 Write 覆盖已有文件(用 Edit)
4. 不改朋友圈素材的序号和内容
5. 不把敏感信息写入 Vault(API key / 凭证 / 密钥保留在本地配置文件)
6. 不批量从冷藏区往主区搬(违背冷藏初衷)
7. 不在主区写入低频原料(推特/公众号自动抓取必须走冷藏区)
8. 不创建无前缀的 04-Concepts 节点
9. **不在 `02-Areas/个人经验/` 用 SOP 化格式写**(v3) — L1 必须保留原汁原味
10. **frontmatter 必须含 layer 字段**(v3) — 缺失视为不合规

### 4.5 自动化脚本写入路径

| 脚本类型 | 写入目标 |
|---------|---------|
| Twitter / X 抓取 | `99-冷藏/推特/<推主名>/` |
| 公众号 RSS | `99-冷藏/公众号博主库/<博主名>/` |
| 播客转录(用户在场) | `07-Archive/播客转录/`(L2,主区) |
| 论文抓取 | `07-Archive/论文原文/`(主区) |
| 朋友圈/活动文案 | `99-冷藏/朋友圈素材/` 或 `99-冷藏/活动文案/` |
| 用户语音转写(原话)(v3) | `02-Areas/个人经验/原话与决策/`(L1) |
| 用户朋友圈本人发布(v3) | `02-Areas/个人经验/原话与决策/`(L1) |

> 新增自动化脚本入库前必须更新此表,默认写入冷藏区(除非来源是用户本人 L1 内容)。

## 五、使用方法:安装/配置/触发

### 5.1 安装

把 `SKILL.md` 和 `PHILOSOPHY.md` 放到 Claude Code 的 skills 目录:

```bash
mkdir -p ~/.claude/skills/obsidian-vault-manager
cp SKILL.md ~/.claude/skills/obsidian-vault-manager/
cp PHILOSOPHY.md ~/.claude/skills/obsidian-vault-manager/
```

### 5.2 配置

修改 `SKILL.md` 中的 `Vault 路径` 占位符,改成你自己的 Obsidian Vault 实际路径:

```
<vault-root>/  →  /path/to/your/obsidian/vault/
```

替换其他占位符(按需):

| 占位符 | 含义 | 替换示例 |
|--------|------|---------|
| `<vault-root>/` | Vault 根路径 | `/Users/me/Documents/MyVault/` |
| `<your-domain>` | 你的领域名 | `创业` / `产品设计` |
| `<personal-experience>` | L1 个人经验目录名 | `个人经验` / `my-voice` |
| `<knowledge-meta>` | 元方法论目录名 | `工具与基建/笔记体系` |
| `<sales-sop>` | 销售 SOP 子目录 | `销售SOP` |
| `<content-sop>` | 内容 SOP 子目录 | `内容制作SOP` |
| `<your-recordings>` | 录音子目录 | `播客转录` |

### 5.3 触发

在 Claude Code 中说以下任一句子,Claude 会自动加载本 skill:

- 「收录到知识库」
- 「存到 Obsidian」
- 「整理笔记」
- 「翻一下推特/公众号有没有 XX」(触发淘金)
- 「补充 Wiki Link」 / 「加链接」 / 「补双向链接」(v3.1)
- 「我刚跟 X 说……」 / 「我感觉……」 / 「今天发生了……」(v3,自动走 L1)

### 5.4 典型使用流程

```
1. 收录: 我把这篇文章存到知识库 → Claude 走收录流程,8 维 wikilink 自动识别
2. 检索: 知识库里有没有 XX 概念 → Claude 走检索三步法
3. 淘金: 翻一下公众号有没有 YY 主题 → Claude 走淘金 SOP
4. 回写: (Claude 给出综合性分析后)→ 自动判断回写时机,落入 04-Concepts
5. L1: 我跟客户王总说了一句话「...」记一下 → Claude 自动走 L1 流程
```

## 六、如何根据你自己的情况改造

这套方法不是"拿来即用"的成品,而是**一套可被改造的骨架**。下面是改造指南。

### 6.1 改造的核心思路

> **不要照搬,要理解 PHILOSOPHY 之后做减法和换名。**

本 Skill 是"已经跑了一年的 5000+ 笔记 vault"提炼出来的具体实现。但你的 vault 大小、领域、自动化流程跟我不同。直接照搬会过度工程。

### 6.2 必改 vs 可改

| 维度 | 必改 | 说明 |
|------|------|------|
| Vault 路径 | ✅ 必改 | `<vault-root>/` 必须改成你自己的路径 |
| 02-Areas 子目录 | ✅ 必改 | 我的领域是 AI/创业/内容,你的可能是设计/学术/法律 |
| 03-People 子目录 | ⚠️ 按需 | 没有客户/同行需求 → 删除整个 03-People |
| 04-Concepts 前缀 | 💡 建议保留 | `[MOC]` `[理论]` `[工程]` `[方法论]` 通用 |
| 高频场景路由表 | ✅ 必改 | 12 场景按你的高频场景调整 |
| 自动抓取流脚本路径 | ⚠️ 按需 | 没有自动抓取流 → 可整个删除 99-冷藏 |
| frontmatter 字段扩展 | 💡 自由 | 按需添加 status / priority 等字段 |

### 6.3 7 个改造维度

#### 维度 1:Vault 路径(必改)

修改 `SKILL.md` 第 30 行附近的 Vault 路径:

```diff
- `<vault-root>/`
+ `/Users/your-username/Documents/MyVault/`
```

同时修改所有 `grep -r "QUERY" <vault-root>/` 模板里的路径。

#### 维度 2:02-Areas 子目录(必改)

把示例的 `<your-domain>` 替换成你自己的领域。建议:

- 单领域用户(只做设计)→ `02-Areas/设计/` 一个目录够了
- 多领域用户 → `02-Areas/<领域A>/` `02-Areas/<领域B>/`...
- 学术研究者 → `02-Areas/<研究方向>/`
- 法律工作者 → `02-Areas/<案件类型>/` + `06-Library/法条/`

#### 维度 3:03-People 子目录(按需)

如果你不做客户工作,可以整个删除 `03-People/`。如果保留,建议按你的人物类型分:

- 销售/咨询 → `客户档案/`、`合作伙伴/`
- 学术 → `导师/`、`合作研究者/`、`引用作者/`
- 内容创作 → `同行案例/`、`粉丝典型/`

#### 维度 4:04-Concepts 前缀(建议保留)

`[MOC]` `[理论]` `[工程]` `[方法论]` 是通用的,建议直接用。如果你想加自己的前缀:

- 学术研究 → `[文献]` `[实验]` `[假说]`
- 设计 → `[案例]` `[原则]` `[组件]`
- 法律 → `[判例]` `[法条]` `[释义]`

但要保持**冗余词黑名单**:调研、报告、深度、完全、详解、全景 — 这些词信息密度为零。

#### 维度 5:高频场景路由表(必改)

把 12 个场景换成你自己的高频场景。逻辑:

1. 列出你过去 30 天最常找资料的 10 个场景
2. 给每个场景写出 2-4 个候选目录(按优先级)
3. 替换 `SKILL.md` 里的高频场景路由表

#### 维度 6:自动抓取流脚本路径(按需)

如果你没有自动抓取流(推特/公众号/RSS):

- 删除 `99-冷藏/` 整个目录
- 删除 `SKILL.md` 里的「冷藏区」「淘金流程」相关章节
- 删除 frontmatter 的 `cherry_picked` 字段
- 4+1 层简化为 4 层(去掉 L5)

如果你有自定义抓取脚本,在「自动化脚本写入路径」表里追加你的脚本类型 + 写入目标。

#### 维度 7:frontmatter 字段扩展(自由)

可按需追加字段:

```yaml
---
status: draft / review / final  # 流转状态
priority: high / medium / low   # 优先级
review_date: YYYY-MM-DD          # 下次复习时间
---
```

但 `layer` 字段是 v3 核心,**不要删**。

### 6.4 改造后验证清单

完成改造后,跑一遍以下验证:

- [ ] Vault 路径占位符全部替换完毕(grep `<vault-root>` 应为 0 命中)

## 七、适用 vs 不适用

### 适用场景

- 上千文件的 Obsidian Vault 维护者
- 有大量自动抓取流(推特/公众号/RSS)淹没主区的用户
- 想给 AI 助手提供高信噪比检索语料的用户
- 已有零散 PARA 实践但缺统一收录/检索 SOP 的用户
- **重视个人经验保鲜的用户(v3)** — 不希望"萃取冲动"磨平一手感受

### 不适用场景

- 文件数 < 200 的小型 Vault(过度工程)
- 只用 Obsidian 做日记/笔记,无自动抓取流的用户
- 不使用 Claude Code 或类似 AI 助手的用户
- 不在乎个人经验"原汁原味"的用户(L1 机制对你没价值)

强行套上反而是负担。
