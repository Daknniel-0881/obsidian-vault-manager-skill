# Obsidian Vault Manager Skill

> A [Claude Code](https://www.anthropic.com/claude-code) skill for managing large Obsidian vaults with PARA + P.A.I.R + Cold Storage architecture.
>
> 一个 Claude Code Skill,用 PARA + P.A.I.R + 冷藏区隔离三套方法论,让上千文件的 Obsidian 知识库保持高信噪比。

---

## 这个 Skill 解决什么问题

当 Obsidian 知识库膨胀到 5000+ 文件后,会出现典型的"知识沼泽"症状:

- 每天自动抓取的推特/公众号原文淹没真正有用的精炼笔记
- 主题分散在十几个目录里,找一个东西要打开 N 个文件夹
- 概念笔记重复,比如"知识库管理"建了三个节点
- 双向链接(Wiki Link)失修,无法把相关知识串起来
- 检索时大量噪声进入上下文,让 AI 助手回答质量下降

**这个 Skill 通过三个核心机制解决以上问题**:

1. **冷藏区隔离**:把高频低密度信息(自动抓取类)放 `99-冷藏/`,主区检索默认豁免
2. **命名前缀规范**:`[MOC]` / `[理论]` / `[工程]` / `[方法论]` 让概念节点一眼分类
3. **淘金机制**:从冷藏区"复制"(不剪切)到主区,源文件加 `cherry_picked` 标记

## 设计哲学

### PARA + P.A.I.R 融合

| 方法论 | 来源 | 核心思想 |
|--------|------|---------|
| **PARA** | Tiago Forte《Building a Second Brain》| Projects / Areas / Resources / Archive 四象限 |
| **P.A.I.R** | flomo / 少楠 | 分类不确定 → 先入 Inbox → 定期清理 → 归位 |

融合后的目录系统:

```
00-Inbox/      ← P.A.I.R 缓冲层
01-Projects/   ← PARA Projects
02-Areas/      ← PARA Areas
03-People/     ← 自定义扩展(人物库)
04-Concepts/   ← 自定义扩展(概念枢纽,Wiki Link 中心)
05-Playbooks/  ← 自定义扩展(SOP/方法论)
06-Library/    ← PARA Resources
07-Archive/    ← PARA Archive
99-冷藏/       ← 冷藏区(默认不检索)
```

### 冷藏区(Cold Storage)隔离原则

**问题**:自动抓取的推特/公众号每天产出几十到几百篇原文,稀释主区信噪比。

**方案**:物理隔离 + 检索豁免 + 淘金机制。

```bash
# 主区检索(默认豁免冷藏)
grep -r "QUERY" <vault-root>/ \
  --exclude-dir=99-冷藏 \
  --exclude-dir=.obsidian

# 淘金检索(显式声明)
grep -r "QUERY" <vault-root>/99-冷藏
```

### 命名前缀规范(04-Concepts)

| 前缀 | 含义 | 示例 |
|------|------|------|
| `[MOC]` | Map of Content,多文件汇总索引 | `[MOC] 用户分层与定价体系.md` |
| `[理论]` | 概念定义、原理、本质 | `[理论] 知识库与RAG完全指南.md` |
| `[工程]` | 具体实现、技巧、最佳实践 | `[工程] 稀缺性推理在上下文工程中的应用.md` |
| `[方法论]` | SOP 之上的元方法论 | `[方法论] 知识库管理.md` |

**冗余词黑名单**(不得用于节点名):调研、报告、深度、完全、详解、全景

**英文节点**:除专有名词(NVIDIA / API / Harness / Hermes / Karpathy)外,全部翻译成中文,专有名词附中文说明。

## 六大核心原则

### 1. 先收再理(P.A.I.R)
分类不确定 → 先扔 `00-Inbox/` → 定期整理归位

### 2. 原料与成品分离(PARA)
- **成品**(02-Areas ~ 06-Library):经过消化提炼的知识
- **高价值原料**(07-Archive 播客/论文):保留在主区
- **低频原料**(99-冷藏):默认不检索,偶尔淘金

### 3. 项目有终点,领域没终点
- `01-Projects/` 完成后 → 方法论提炼到 `05-Playbooks/`,记录归档到 `07-Archive/已结案项目/`
- `02-Areas/` 只做加法,持续积累

### 4. 原文神圣不可侵犯
- 收录原文一字不改
- 助手分析追加在原文之后的独立章节
- 朋友圈素材只读不写,序号不动

### 5. 复利优先
- 每次入库都要思考:这个知识能不能复用?
- 能提炼为 SOP/方法论 → 同步写入 `05-Playbooks/`
- 能抽象为概念 → 同步创建/更新 `04-Concepts/` 节点

### 6. 冷藏不删除
- 99-冷藏区是隔离不是丢弃
- 冷藏区文件原文一字不改
- 高价值发现 → 复制(不剪切)到主区,原文件加 `cherry_picked: true`

## 工作流程

### 收录流程

```
内容 → 类型判断 → 选目录 → 创建笔记(frontmatter + 正文 + 助手分析 + Wiki Link) → 更新 MOC
```

**Step 1:类型判断 → 目录路由**

```
内容是什么?
├── 具体项目交付物 → 01-Projects/
├── 某领域的深度内容 → 02-Areas/<领域>/
├── 关于某个人的深度拆解 → 03-People/
├── 可跨域复用的核心概念 → 04-Concepts/(必带 [前缀])
├── 可复用的方法论/SOP → 05-Playbooks/
├── 工具/资源/失败案例/论文/网络/课程 → 06-Library/<子目录>/
├── 已结案项目 / 高价值原料(播客/论文)→ 07-Archive/<子目录>/
├── 推特/公众号/朋友圈/活动文案 → 99-冷藏/<子目录>/
└── 不确定 → 00-Inbox/
```

**Step 2:创建笔记**

```markdown
---
tags: [主标签, 辅助标签]
created: YYYY-MM-DD
source: <来源URL或说明>
author: <作者>
cherry_picked: false  # 仅冷藏区淘金后才设 true
---

# 标题

> 一句话摘要

## 正文内容

(原文完整保留)

## AI 助手分析与梳理

(助手的补充分析,与原文明确分隔)

## 相关链接

- [[相关概念1]]
- [[相关文档2]]
```

**Step 3:建立 Wiki Link**

每篇精炼笔记必须建立三类链接:

1. **向上链接**:链接到所属领域的 MOC(`_MOC-*.md`)
2. **横向链接**:链接到相关概念节点(`04-Concepts/`)
3. **向下链接**:链接到引用的具体文档

### 检索流程(三步法)

1. **目录路由**:根据问题主题匹配目录(参考路由表)
2. **关键词 Grep**:在匹配目录内搜,默认豁免冷藏
3. **Wiki Link 跳转**:读取 `[[]]` 链接追溯关联知识

```bash
# 主区默认检索(豁免冷藏 + 系统目录)
grep -r "QUERY" <vault-root>/ \
  --exclude-dir=99-冷藏 \
  --exclude-dir=_system \
  --exclude-dir=.obsidian \
  --exclude-dir=.smart-env \
  --exclude-dir=Assets \
  --exclude-dir=Excalidraw

# 淘金检索(显式声明,仅冷藏)
grep -r "QUERY" <vault-root>/99-冷藏
```

### 淘金流程(冷藏区 → 主区)

**触发时机**:
- 助手判断某主题在主区无果,但冷藏区可能有
- 用户明确说"翻一下推特/公众号有没有 XX"

**3 步 SOP**:

1. **定位**:在 `99-冷藏/` 内显式 grep 关键词
2. **评估**:核对目标文件 frontmatter 的 source/author/date,判断是否值得提炼
3. **迁移**:
   - **复制**(不是 mv)到 02-Areas 对应子目录
   - 原冷藏文件 frontmatter 加 `cherry_picked: true`
   - 主区版本加 Wiki Link 反向指向 `99-冷藏/原文件`
   - 主区版本添加「助手分析」章节

**淘金禁忌**:
- 不批量从冷藏区往主区搬(违背冷藏初衷)
- 同一主题已有主区文件 → 在已有文件追加引用,不新建

### 整理流程

| 频率 | 任务 |
|------|------|
| 每周 | 清理 `00-Inbox/`(归位 → 移动 → 更新 MOC) |
| 每季度 | 冷藏区淘金(列高价值标签 → 评估 → 走淘金 SOP) |
| 持续 | Archive 提炼(高价值原料 → 02-Areas/05-Playbooks 精炼笔记) |

## 禁止事项

1. **不删除任何文件**(除非用户明确要求)
2. **不修改原文内容**(只追加,原文一字不改)
3. **不用 Write 覆盖已有文件**(用 Edit)
4. **不改朋友圈素材的序号和内容**
5. **不把敏感信息写入 Vault**(API key / 凭证 / 密钥保留在本地配置文件)
6. **不批量从冷藏区往主区搬**(违背冷藏初衷)
7. **不在主区写入低频原料**(推特/公众号自动抓取必须走冷藏区)
8. **不创建无前缀的 04-Concepts 节点**

## 安装与使用

### 1. 安装

把 `SKILL.md` 放到 Claude Code 的 skills 目录:

```bash
mkdir -p ~/.claude/skills/obsidian-vault-manager
cp SKILL.md ~/.claude/skills/obsidian-vault-manager/
```

### 2. 配置

修改 `SKILL.md` 中的 `Vault 路径` 占位符,改成你自己的 Obsidian Vault 实际路径:

```
<vault-root>/  →  /path/to/your/obsidian/vault/
```

### 3. 触发

在 Claude Code 中说以下任一句子,Claude 会自动加载本 skill:

- 「收录到知识库」
- 「存到 Obsidian」
- 「整理笔记」
- 「翻一下推特/公众号有没有 XX」(触发淘金)
- 「补充 Wiki Link」

### 4. 自定义

按你自己的领域调整 `02-Areas/` 子目录:

```
02-Areas/
├── 你的领域A/
├── 你的领域B/
└── ...
```

`04-Concepts/` 节点按 `[前缀]` 规范创建即可。

## 适用场景

- 上千文件的 Obsidian Vault 维护者
- 有大量自动抓取流(推特/公众号/RSS)淹没主区的用户
- 想给 AI 助手提供高信噪比检索语料的用户
- 已有零散 PARA 实践但缺统一收录/检索 SOP 的用户

## 不适用场景

- 文件数 < 200 的小型 Vault(过度工程)
- 只用 Obsidian 做日记/笔记,无自动抓取流的用户
- 不使用 Claude Code 或类似 AI 助手的用户
