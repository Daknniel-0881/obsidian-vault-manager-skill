## 二、4+1 层信息分层模型(本 Skill 的核心)

### 2.1 为什么分层

AI 上下文越来越大,大家觉得"不用整理笔记了反正都能塞进去"——这是错的。

**真正稀缺的不是上下文长度,是上下文质量**。当一个 vault 里同时存在:
- 你自己的原话/亲历(独一无二,不可再生)
- 你在场录制的会议/对话原文(中等独特性)
- 你从外部抓的推特/文章(网上随处可见)
- 你萃取过的方法论成品(高密度)

把它们扔在一起,AI 检索时会用网上随处可见的低密度原料淹没你独一无二的一手感受。**分层 = 信噪比治理**。

### 2.2 五层定义

| 层 | 名称 | 定义 | 物理目录 | frontmatter |
|---|------|-----|---------|-------------|
| **L1** | 个人经验层 | 你自己的原话、亲历、一手感受、带个体烙印的判断 | `02-Areas/<personal-experience>/` | `layer: experience` |
| **L2** | 在场素材层 | 你自己在场的录音、会议、对话原文(中等独特性) | `07-Archive/播客转录/`、`02-Areas/<your-recordings>/` | `layer: scene` |
| **L3** | 外部入口层 | 收件箱、未分类、未消化原料 | `00-Inbox/` | `layer: inbox` |
| **L4** | 萃取成品层 | 加工过的成品/半成品(SOP、概念、方法论) | `02-Areas/`、`04-Concepts/`、`05-Playbooks/`、`06-Library/`、`07-Archive/已结案项目/` | `layer: extract` |
| **L5** | 冷藏原料层 | 不删除但不主动检索的低频原料(本 Skill 在原 4 层基础上的扩展) | `99-冷藏/` | `layer: cold` |

### 2.3 为什么有 L5(冷藏层)

余一原版只有 4 层。本 Skill 加了 L5,因为有自动抓取流(推特/公众号/RSS)的用户,每天会涌入几十到几百篇原文,这些既不是 L1(不是你的)、也不是 L4(没萃取),如果扔进 L3(Inbox)会瞬间淹没真正需要处理的内容。

**L5 = 隔离不是丢弃**。冷藏区文件原文一字不改,默认不检索,但留着以便淘金(详见 SKILL.md 淘金 SOP)。

### 2.4 关键判断:L1 vs L4

最容易混淆的是「个人经验」(L1) vs「萃取成品」(L4)。

判断标准 = **这一篇是事件本身的原始记录,还是为了萃取出可复用的 SOP?**

| | L1 个人经验 | L4 萃取成品 |
|--|------------|-----------|
| 加工度 | 原汁原味 | 高度抽象 |
| 时态 | "那一刻"的感受 | "下次该怎么做" |
| 文体 | 散文/碎片/独白 | SOP/Checklist/方法论 |
| 例子 | 「今天客户说 XX,我当时...」 | 「客户对话 SOP:第一步...」 |
| 目录 | `02-Areas/<personal-experience>/` | `05-Playbooks/<your-domain>/` |

**为什么这个区分重要**:L1 是 AI 时代你**最稀缺**的资产——你的口吻、你的直觉、你的犹豫,是 AI 永远学不会的"你"。一旦塞进 SOP 萃取流程,就被磨平了。所以**用目录物理隔离**,不让"萃取冲动"污染原汁原味的 L1。

## 三、六大设计原则

### 1. 先收再理(P.A.I.R)
分类不确定 → 先扔 `00-Inbox/` → 定期归位。**不要因为纠结分类而耽误收录**。

### 2. 原料与成品分离(PARA + 4+1 层)
- L1 / L2 / L5 是原料层(不萃取,只保留)
- L4 是成品层(高度加工)
- L3 是缓冲层(等待分类)

**原文神圣不可侵犯**:原文一字不改,助手分析追加在独立章节。

### 3. 项目有终点,领域没终点
- `01-Projects/` 完成后 → 方法论提炼到 `05-Playbooks/`,记录归档到 `07-Archive/已结案项目/`
- `02-Areas/` 只做加法,持续积累

### 4. 复利优先
每次入库都问:**这个知识能不能复用?**
- 能提炼为 SOP/方法论 → 同步写入 `05-Playbooks/`
- 能抽象为概念 → 同步创建/更新 `04-Concepts/` 节点

### 5. LLM 维护、人审核
对标 Karpathy 的 LLM Wiki 模式:
- AI 助手做苦力(摘要、交叉引用、归档、Wiki Link 维护)
- 人把关方向(决定什么进 L1、什么算 L4 成品、什么进冷藏)
- AI 生成内容明确标注来源(`> 来源:AI 助手分析(YYYY-MM-DD)`),与原文物理分隔

### 6. 冷藏不删除
- 99-冷藏区是隔离不是丢弃
- 冷藏区文件原文一字不改
- 高价值发现 → **复制**(不剪切)到主区,源文件加 `cherry_picked: true`

## 四、命名前缀规范的设计思想

`04-Concepts/` 是知识网络的十字路口,所有跨域知识在这里汇聚。前缀规范的目的是 **一眼分类**:

| 前缀 | 含义 | 思想 |
|------|------|------|
| `[MOC]` | Map of Content,索引页 | 这是"导航",不是"内容" |
| `[理论]` | 概念定义、原理、本质 | 这是"是什么",回答存在性问题 |
| `[工程]` | 具体实现、技巧 | 这是"怎么做",回答方法问题 |
| `[方法论]` | SOP 之上的元方法论 | 这是"怎么思考",回答元问题 |

**冗余词黑名单**(节点名禁止使用):调研、报告、深度、完全、详解、全景。

理由:这些词都是"凑字数"的,信息密度为零,加了反而让节点名失焦。一篇好的概念笔记应该用最少的词点穿主题。

## 五、与 Karpathy LLM Wiki 方案的对照

Karpathy 提出的是**通用抽象框架**,本 Skill 是**已经跑了一段时间的具体实现**。

| Karpathy 概念 | 本 Skill 对应物 |
|---------------|---------------|
| `raw/` | `07-Archive/` + `99-冷藏/` |
| `wiki/` | `02-Areas` ~ `06-Library` |
| Schema (CLAUDE.md) | 本 Skill SKILL.md |
| `index.md` | `_MOC-*.md`(各目录 MOC) |
| `log.md` | `_system/_log/` 操作日志 |
| Ingest | 收录流程(类型判断 → 选目录 → frontmatter → Wiki Link → 更新 MOC) |
| Query | 检索三步法 + 查询结果回写 |
| Lint | 健康检查清单(孤立页面 / 缺失概念 / 过期信息 / 矛盾标记 / 断链 / MOC 遗漏) |

**比 Karpathy 多的**:
- `03-People/` 人物库(同行 / 客户 / 导师 / 合作伙伴)
- `04-Concepts/` 跨域概念枢纽(他的概念分散在 wiki 中)
- 4+1 层信息分层模型(他没做信息源头的污染治理)
- 99-冷藏区(他没考虑高频抓取流)

**从 Karpathy 学的**:
- 知识是 LLM 编译产物的思路
- 操作日志的重要性
- 查询结果回写习惯(好回答存回 wiki)

## 五·补、与 kytmanov olw(obsidian-llm-wiki-local)方案的对照(v3.2)

Karpathy 给的是**抽象框架**,kytmanov 给的是**第一个真实跑起来的 CLI 工具**(Python + Click,带 doctor / ingest / compile / review / approve / reject / unblock 子命令)。本 Skill v3.2 借鉴了它的工程化思想,但**继续保留无脚本的纯文档可执行规范形态**(让任何 AI 助手能直接读 SKILL.md 跑流程,不依赖额外 CLI)。

| olw 概念 | 本 Skill v3.2 对应物 |
|----------|---------------------|
| `raw/` vs `wiki/` 物理双区 | L2/L5(原料)vs L4(萃取),用前缀 + 目录区分 |
| `olw ingest` | 收录流程 + frontmatter 写齐 + 内容指纹 dedup |
| `olw compile` | 概念提炼管道 5 步 SOP(Ingest→Validate→Match→Draft→Review) |
| `olw review/approve/reject` | Step 5 人审 + rejected_drafts 字段 |
| `olw unblock` | 用户手动从 `_system/_blocked.md` 移除条目 |
| 5 次驳回自动 block | Rejection Feedback 反馈闭环 |
| frontmatter `aliases:` | Aliases 别名机制 + `_system/_aliases.md` 全局表 |
| 自动建 stub 概念 | Stubs 桩节点机制(`04-Concepts/_stubs/`) |
| Hand-edit detection 跳过覆写 | Hand-edit Protection 第 7 大原则 + AI 区块标记 |
| frontmatter `confidence:` / `status:` | v3.2 强制字段 |

**比 olw 多的**:
- 4+1 层信息分层(olw 只有 raw/wiki 二分,没有信息源头治理)
- L1 个人经验层(olw 不区分"原话"和"成品",会被 SOP 萃取流程吞没)
- 99-冷藏区(olw 没考虑高频抓取流)
- 高频场景路由表(12 场景的快速定位)
- 中文优先 + 命名前缀规范([MOC]/[理论]/[工程]/[方法论])

**从 olw 学的**:
- 概念提炼必须**工程化**(不能靠 AI 拍脑袋判断该不该建节点)
- 别名机制是**减少死链的核心**(不只是 wikilink 自动识别要查别名表)
- 桩节点是**承认"我现在没空写完"的工程优雅**(占位 + 提升规则)
- 驳回反馈闭环防止**AI 反复浪费 token 在已被人否定的提炼上**
- Hand-edit Protection 是**人机协作长期可持续的前提**(没有它,AI 会反复覆盖人改的内容,用户最终会放弃 AI 维护)

## 六、什么时候不该用本 Skill

诚实的边界:

- **文件数 < 200 的小型 Vault**:过度工程,简单分类够用
- **只用 Obsidian 写日记/碎碎念**:没有萃取需求,L4 层用不上
- **没有自动抓取流的用户**:99-冷藏区是空的,L5 层无意义
- **不使用 AI 助手 / Claude Code**:很多机制(自动入库、Wiki Link 补充、淘金)失去自动化价值

强行套上反而是负担。

## 七、长期方向

| 阶段 | 方向 | 触发条件 |
|------|------|---------|
| 近期 | 自动 Lint 脚本(孤立页面检测、断链修复) | 知识库 > 500 篇 |
| 中期 | 本地 RAG / 向量索引(grep + Wiki Link 不够用时) | 知识库 > 3000 篇 + 跨域查询频繁 |
| 远期 | 合成数据 + 微调(领域知识足够深) | 形成稳定的"个人知识图谱"且有微调资源 |

短期不要盲目上 RAG。Karpathy 和很多实践者都验证过:中等规模(< 3000 篇)下 grep + Wiki Link + 目录路由够用,RAG 反而引入幻觉风险。
